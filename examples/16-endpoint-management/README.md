# Chapter 16: Endpoint Management

Demonstrates four endpoint management patterns with Apache Camel on Quarkus:

- **Messaging Gateway** — wraps `FluentProducerTemplate` behind a domain-specific `OrderMessagingGateway` bean so application code never touches messaging APIs directly
- **Selective Consumer** — Kafka consumer that filters messages by the `containsHazmat` header, accepting only non-hazmat orders
- **Channel Purger** — discards stale messages older than a configurable cutoff, forwarding only recent messages to a clean topic
- **Messaging Mapper** — maps between JSON wire format and typed `Order` objects using `unmarshal().json(Order.class)` and a `@ApplicationScoped` `OrderService` bean

## Prerequisites

Start the infrastructure stack:

```bash
cd examples/_infra
./../../scripts/setup-stack.sh
```

## Running

```bash
cd examples/16-endpoint-management
mvn quarkus:dev
```

The demo data generator creates a new order every 5 seconds with varying attributes (some hazmat, some with stale timestamps). The gateway demo timer publishes an additional order every 8 seconds through the messaging gateway.

## Kafka topics

| Topic | Description |
|-------|-------------|
| `eip.orders.placed` | All incoming orders (generated by demo + gateway) |
| `eip.orders.inventory-request` | Inventory check requests via the gateway |
| `eip.orders.accepted` | Non-hazmat orders accepted by the selective consumer |
| `eip.orders.clean` | Recent messages that passed the channel purger |

## Verification

Open Kafka UI at [http://localhost:8080](http://localhost:8080) and observe:

1. Orders appearing on `eip.orders.placed` from both the demo generator and the gateway
2. Only non-hazmat orders forwarded to `eip.orders.accepted`
3. Stale messages purged; recent messages forwarded to `eip.orders.clean`
4. Mapped `Order` objects processed by `OrderService` in the application logs
