# Chapter 18: Testing and Management

Demonstrates four testing and management patterns with Apache Camel on Quarkus:

- **Test Message** — injects synthetic test orders (negative IDs, `test_message=true` header) every 30 seconds; a verifier route checks processed output and logs pass/fail; the business route filters test messages before processing
- **Detour** — conditionally bypasses an enrichment step based on the `feature.enrichment.enabled` property
- **Smart Proxy** — routes payment requests to a mock or production gateway based on `payment.gateway.mode`
- **Managed Channel Adapter (Circuit Breaker)** — wraps an external inventory call in a circuit breaker; when the service fails, the fallback routes to a DLQ

## Prerequisites

Start the infrastructure stack:

```bash
cd examples/_infra
./../../scripts/setup-stack.sh
```

## Running

```bash
cd examples/18-testing-management
mvn quarkus:dev
```

The demo data generator creates a new order every 5 seconds and publishes to `eip.orders.placed`. Test messages are injected every 30 seconds with negative order IDs.

## Testing the Smart Proxy

```bash
# Mock mode (default)
curl -X POST http://localhost:8080/payments/process \
  -H "Content-Type: application/json" \
  -d '{"order_id": 42, "amount": 150.00}'
```

## Configuration

| Property | Default | Description |
|----------|---------|-------------|
| `feature.enrichment.enabled` | `true` | Detour: enable/disable enrichment step |
| `payment.gateway.mode` | `mock` | Smart Proxy: `mock` or `production` |

## Kafka topics

| Topic | Description |
|-------|-------------|
| `eip.orders.placed` | All incoming orders (generated by demo + test messages) |
| `eip.orders.processed` | Real orders after test message filtering |
| `eip.orders.enriched` | Orders after detour/enrichment step |
| `eip.orders.inventory-checked` | Orders that passed inventory check |
| `eip.orders.dlq` | Orders that failed circuit breaker (fallback) |
| `eip.test.failures` | Test messages that did not match expectations |

## Verification

Open Kafka UI at [http://localhost:8080](http://localhost:8080) and observe:

1. Test messages with negative IDs flowing through the verifier
2. Real orders passing through the business processor
3. Orders routed through or around enrichment based on the detour flag
4. DLQ messages when the circuit breaker opens
